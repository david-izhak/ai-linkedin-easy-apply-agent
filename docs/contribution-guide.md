# Руководство по контрибьюции

Мы приветствуем вклад в развитие проекта! Это руководство поможет вам понять, как вы можете улучшить **LinkedIn Easy Apply Bot**.

## Стиль кода (Code Style)

Проект придерживается стандарта **PEP 8** и использует **Ruff** для автоматической проверки и форматирования кода.

- **Проверка:** Перед созданием Pull Request убедитесь, что ваш код проходит проверку:
  ```bash
  uv run ruff check .
  ```

- **Форматирование:**
  ```bash
  uv run ruff format .
  ```

## Как добавить новый тип поля в форму

Формы "Easy Apply" могут содержать разнообразные типы полей. Если вы столкнулись с новым, неизвестным боту полем, вы можете добавить его поддержку.

1.  **Создайте новый модуль:** В директории `apply_form/` создайте новый файл, например, `fill_new_field_type.py`.
2.  **Реализуйте логику:** В этом файле напишите функцию, которая принимает объект `playwright.Page` и данные из `config.py` и выполняет необходимые действия для заполнения этого поля.
3.  **Интегрируйте в оркестратор:** Откройте `apply_form/fill_fields.py`.
    - Импортируйте вашу новую функцию.
    - В цикле обработки полей добавьте условие, которое будет определять ваш новый тип поля (например, по тексту `label` или CSS-классу) и вызывать вашу функцию.
4.  **Добавьте тесты:** Создайте unit-тесты в `tests/unit/apply_form/` для проверки логики вашего нового модуля.

## Как добавить поддержку нового LLM-провайдера

Модуль `llm/` спроектирован с учетом расширяемости.

1.  **Установите клиентскую библиотеку:** Добавьте в `pyproject.toml` новую зависимость, используя `uv`:
    ```bash
    uv add langchain-gemini  # пример
    ```
2.  **Расширьте `client_factory.py`:**
    - Откройте `llm/client_factory.py`.
    - Добавьте новую ветку для вашего провайдера в функцию создания клиента.
    - Инициализируйте клиент для нового провайдера, используя соответствующий класс из `langchain`.
3.  **Обновите конфигурацию:**
    - В `llm/config.py` добавьте нового провайдера в перечисление для поля `llm_provider`.
    - Убедитесь, что все необходимые параметры (например, API-ключ) обрабатываются корректно.
4.  **Добавьте тесты:** Создайте unit-тесты в `tests/unit/llm/` для проверки инициализации нового провайдера.

## Процесс внесения изменений (Pull Request Workflow)

1.  **Создайте Fork:** Сделайте форк основного репозитория.
2.  **Создайте ветку:** Для каждой новой фичи или исправления создавайте отдельную ветку.
    ```bash
    git checkout -b feature/my-new-feature
    ```
3.  **Внесите изменения:** Напишите код и **обязательно добавьте тесты**, покрывающие ваши изменения.
4.  **Проверьте код:** 
    - Убедитесь, что код проходит проверку Ruff:
      ```bash
      uv run ruff check .
      uv run ruff format .
      ```
    - Запустите все тесты:
      ```bash
      uv run pytest tests/ -v
      ```
5.  **Создайте Pull Request (PR):**
    - Отправьте вашу ветку в ваш форк.
    - Создайте Pull Request в основной репозиторий.
    - В описании PR подробно опишите, какие изменения вы внесли и зачем.
6.  **Пройдите Code Review:** Будьте готовы к обсуждению и внесению правок по результатам ревью.
